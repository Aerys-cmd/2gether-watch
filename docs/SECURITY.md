# Security

This document describes the security model of 2gether Watch, known
considerations, and guidance for self-hosters.

---

## Security model

### Signaling server is a relay only

The ASP.NET Core backend is a **pure WebSocket signaling relay**. It:

- assigns each connecting client a random 12-hex-character peer ID
- forwards JSON messages (WebRTC offers/answers/ICE candidates, chat, video sync)
  between peers inside a room
- injects an authoritative `"from"` field into every relayed message so peers
  cannot spoof the sender identity
- drops any non-JSON message sent after the initial handshake to prevent clients
  from injecting server control frames (e.g. `peer-left:`, `peer-joined:`)

**No audio, video, or screen-share content ever passes through the server.**
All media travels peer-to-peer over encrypted WebRTC DTLS/SRTP channels.

### Room privacy

Rooms are identified by a random ID. There is **no authentication** — anyone
who knows the room ID can join. Room IDs generated by the app use
`crypto.randomUUID()` (48-bit first segment, ~281 trillion possibilities), making
enumeration impractical.

- **Share room links only with people you trust.**
- Rooms are ephemeral: they exist in memory only while at least one peer is
  connected and are destroyed when the last peer leaves.
- The server enforces a maximum of **10 peers per room**.

### Transport security

The hosted instance at `2gether-watch.lunavex.com` enforces:

- **HTTPS everywhere** — HTTP is redirected to HTTPS by Traefik
- **HSTS** — browsers will refuse plain HTTP connections for 30 days after the
  first visit
- **WebSocket Secure (WSS)** — the client automatically uses `wss://` on HTTPS
  origins

### Input validation

| Input | Server-side validation |
|---|---|
| Room ID (HTTP route) | Regex `^[A-Za-z0-9_-]{1,64}$`; invalid IDs redirect to home |
| Room ID (WebSocket `join:` message) | Same regex; invalid IDs close the connection |
| Relay message size | Max **64 KB** per message; larger payloads close the connection |
| Video URLs (sync) | Must parse as valid `http:` or `https:` URLs (validated client-side before relaying and before applying) |
| Chat messages | Max **500 characters** (client-side `maxlength`); rendered with `textContent` (XSS-safe) |

### WebRTC security

- All peer-to-peer streams are encrypted via **DTLS 1.2/1.3 + SRTP** by the
  browser's WebRTC implementation.
- ICE candidates are exchanged only through the signaling server; the server
  cannot read the media payload.
- Only **STUN** servers are configured by default. STUN does not relay traffic —
  it only helps discover public IP addresses. No media is routed through STUN.

## Known considerations

### No server-side rate limiting

The server does not currently enforce per-IP connection limits or message-rate
limits on the WebSocket endpoint. A single client could open many connections or
send many messages quickly.

**Mitigation for self-hosters**: place a reverse proxy (Traefik, nginx) in front
and configure connection / request rate limiting there.

### TURN server not included

Without a TURN relay, peers behind strict NAT (e.g. symmetric NAT in some
corporate networks) may fail to connect. The app will show a connection error in
those cases. No fallback is attempted.

**Mitigation**: add a TURN server as described in
[docs/SELF-HOSTING.md](SELF-HOSTING.md#adding-a-turn-server-optional).

### STUN exposes public IPs

STUN-based ICE gathers the real public IP addresses of peers and shares them
as ICE candidates via the signaling server. This is how peer-to-peer connections
work, but it means participants can learn each other's IP addresses.

**Mitigation**: if IP privacy is critical, deploy a TURN server and configure
`iceTransportPolicy: "relay"` in `webrtc.js` to force all traffic through the
relay. Be aware this removes the P2P benefit.

### No Content-Security-Policy header

The app currently does not set a `Content-Security-Policy` response header.

### Usernames are self-reported

Display names are chosen by the user and stored in `localStorage`. They are
displayed via `textContent` (XSS-safe) but are not authenticated. Any participant
can call themselves any name.

### YouTube iframe API

When a YouTube URL is loaded, the YouTube IFrame Player API script
(`youtube.com/iframe_api`) is fetched from Google. This is standard YouTube embed
behaviour and means Google's usual privacy policies apply for that content.

## Responsible disclosure

If you discover a security vulnerability in this project, please report it
privately by opening a
[GitHub Security Advisory](https://github.com/Aerys-cmd/2gether-watch/security/advisories/new)
rather than a public issue. We will acknowledge and address confirmed
vulnerabilities as quickly as possible.

---

## Checklist for self-hosters

- [ ] Serve the app over **HTTPS** with a valid certificate
- [ ] Enable **HSTS** at the reverse proxy or in `appsettings.json`
- [ ] Configure **rate limiting** at the reverse proxy for `/ws`
- [ ] Optionally deploy a **TURN server** for users behind strict NAT
- [ ] Review the `AllowedHosts` setting in `appsettings.json` and restrict it
      to your domain in production
- [ ] Keep the Docker image (or .NET runtime) up to date for security patches
